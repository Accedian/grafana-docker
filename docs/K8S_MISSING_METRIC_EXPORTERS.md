# Missing Metric Exporters on K8s Deployments

**Date**: 2025-05-21
**Status**: For Team Review
**Impact**: Affects ALL K8s deployments — these are systemic gaps from the Docker Swarm → K8s migration, not deployment-specific issues.

## Summary

Several ops dashboards reference Prometheus metrics that are not being collected on K8s deployments because the corresponding exporters or scrape configurations were not ported from the Docker Swarm setup. The old Docker Swarm architecture relied on a `ramen` service and inline Prometheus scrape configs generated by `generate_prometheus_config.sh`, none of which exist in the K8s helm charts.

---

## 1. Blackbox Exporter — Certificate Expiry Monitoring

### Affected Dashboards/Widgets
- `ops-days_until_cert_expire` — "# of days until cert expires"
- `ops-deployment_all` — "1 Day Cert Expiry" widget

### Missing Metric
`probe_ssl_earliest_cert_expiry`

### Current State
- Blackbox Exporter **IS deployed** via `grafana-docker/helm/templates/blackbox.yaml`
- A ClusterIP Service exists at `grafana-docker/helm/templates/service.yaml` (port 9115)
- **But**: no ServiceMonitor, no Probe CRD, and no `blackbox.yml` ConfigMap with probe module definitions
- Prometheus has no scrape config targeting the blackbox exporter
- No HTTPS probe targets are configured anywhere
- Only `prober_probe_*` metrics exist (from kubelet probes), not `probe_*` from blackbox

### What's Needed
1. A ConfigMap with `blackbox.yml` defining at least an `http_2xx` module with `tls_config` for HTTPS probing
2. A Prometheus `Probe` CRD or `additionalScrapeConfig` that:
   - Targets the blackbox exporter service
   - Specifies HTTPS endpoints to probe (deployment domain, API endpoints, etc.)
   - Uses the `__param_target` relabeling pattern standard for blackbox
3. Alternatively, a ServiceMonitor if the blackbox exporter exposes its own metrics

### Effort Estimate
Medium — the exporter is already running. Needs config + scrape plumbing.

## 2. Druid Full Metrics Suite

### Affected Dashboards/Widgets
- `ops-datasource_availability` — "DataSource Availability alert" (`druid_coordinator_datasourceLoadStatus`)
- `ops-deployment_all`:
  - "Druid Status check" (`druid_status_gauge`) — shows ERROR
  - "Druid's overlord view" (`druid_status_gauge` - `druid_server_gauge`)
  - Any widget using `druid_coordinator_*` or `druid_segment_*` metrics

### Missing Metrics
- `druid_status_gauge`
- `druid_server_gauge`
- `druid_segment_count`
- `druid_coordinator_datasourceLoadStatus`

### Current State
- Only 3 basic Druid metrics exist: `druid_ingestion_aggregateLag`, `druid_ingestion_taskLast24Hour`, `druid_ingestion_taskTotals`
- The Docker Swarm setup used `ramen` services (ports 9800-9809) to expose comprehensive Druid metrics, with scrape targets generated by `aod-deployer/files/ramen/generate_prometheus_config.sh`
- That script uses `docker stack ps` — Docker Swarm only, not applicable to K8s
- No K8s equivalent exporter or ServiceMonitor exists for Druid coordinator/status/segment metrics

### What's Needed
1. Determine which service currently provides the 3 existing `druid_ingestion_*` metrics and whether it can be extended
2. Deploy a Druid Prometheus exporter sidecar or standalone service that scrapes all Druid component APIs:
   - Coordinator: `/status`, datasource load status
   - Historical/Broker/MiddleManager/Overlord: status gauges
   - Segments: count, size per datasource
3. Create a ServiceMonitor to register the exporter with Prometheus

### Effort Estimate
High — requires understanding the ramen service functionality and building a K8s-native replacement or adapting the existing metrics source.

## 3. Backup Metrics Exporter

### Affected Dashboards/Widgets
- `ops-deployment_all`:
  - "Size of Last Registered backup" (`skylight_backups_size_of_last_backups`)
  - "Number of backups stored" (`skylight_backups_number_of_backups`)

### Missing Metrics
- `skylight_backups_time_since_last_backup`
- `skylight_backups_size_of_last_backups`
- `skylight_backups_number_of_backups`

### Current State
- Backup CronJobs ARE running and completing successfully:
  - `backup-create-couchdb`
  - `backup-create-elasticsearch`
  - `postgres-backup-create`
  - `yang-postgres-backup-create`
- **Zero** `skylight_backups_*` metrics exist in Prometheus
- No backup metrics exporter exists anywhere in `aod-deployer` or `grafana-docker` helm charts
- These metrics were likely produced by the `ramen` service in the Docker Swarm setup
- No reference to `skylight_backups` exists in any deployer configuration

### What's Needed
1. A lightweight Prometheus exporter (Deployment or CronJob) that:
   - Queries backup storage (S3/GCS bucket) for backup metadata
   - Exposes `skylight_backups_time_since_last_backup` (seconds since last successful backup)
   - Exposes `skylight_backups_size_of_last_backups` (bytes)
   - Exposes `skylight_backups_number_of_backups` (count of stored backups)
2. A ServiceMonitor to register the exporter with Prometheus

### Effort Estimate
Medium — well-defined scope, needs a simple exporter that checks backup state.

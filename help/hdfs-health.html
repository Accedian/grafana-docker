HDFS & Hadoop Health Alert Guide

Troubleshooting distributed file system integrity and node health

Affected alerts

HDFS Corrupt Blocks HDFS Under-Replicated Blocks HDFS Time Since Last Checkpoint Hadoop Dead Nodes

What do these alerts mean?

The platform uses HDFS (Hadoop Distributed File System) to store ingested data and intermediate processing results. These alerts monitor the health of the HDFS cluster:

  • Corrupt blocks — data blocks that cannot be read correctly, indicating potential data integrity issues on one or more storage nodes.
  • Under-replicated blocks — blocks that do not have the configured number of copies. A sustained high count may signal node failures or capacity constraints.
  • Checkpoint delay — the NameNode has not completed a filesystem checkpoint within the expected interval. Checkpoints protect against data loss on NameNode failure.
  • Dead nodes — one or more DataNodes are no longer responding. Data stored on dead nodes becomes temporarily unavailable until the node recovers or blocks are re-replicated.

Impact

HDFS health issues can affect data availability and processing:

  • Ingestion jobs may fail if they cannot write to HDFS.
  • Queries and batch processing may return incomplete results if required blocks are corrupt or unavailable.
  • Prolonged node failures reduce cluster capacity and increase the risk of data loss if additional nodes fail before re-replication completes.

Investigation steps

  1. 1 Open the alert in Grafana and review the metric values to assess severity — a small number of under-replicated blocks may be transient, while a growing count indicates a real problem.
  2. 2 Check the NameNode status and logs for errors related to block replication, checkpoint failures, or unresponsive DataNodes.
  3. 3 Identify any dead or decommissioning DataNodes and verify whether they are experiencing hardware issues, disk failures, or network connectivity problems.
  4. 4 Review disk utilization on all DataNodes — HDFS nodes running low on disk space will stop accepting new blocks, which can cause replication and write failures.

Remediation

Corrupt blocks

Identify the affected files and check whether the data can be regenerated by re-running the relevant ingestion or processing jobs. If the corruption is caused by failing disks, replace the affected storage and let HDFS re-replicate.

Under-replicated blocks

Under-replicated blocks usually self-heal once dead nodes are restored or new capacity is added. Monitor the metric to confirm the count is decreasing. If it remains high, check whether DataNode resources (disk, network) are constraining replication.

Checkpoint delay

Verify that the secondary NameNode (or standby NameNode) is healthy and has sufficient disk space and memory. Restart the checkpoint process if it has stalled. Extended checkpoint delays increase the recovery time in the event of a NameNode failure.

Dead nodes

Investigate the root cause (hardware failure, network partition, process crash) and bring the node back online. HDFS will automatically begin re-replicating blocks once a node is confirmed dead, but restoring the original node avoids unnecessary data movement.

Verification

  • Confirm the corrupt block and under-replicated block counts are returning to zero.
  • Verify all DataNodes are reporting as live and healthy.
  • Check that the most recent checkpoint timestamp is within the expected interval.
  • Wait for the next Grafana evaluation cycle and confirm alerts return to Normal.

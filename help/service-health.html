Application Service Health Alert Guide

Troubleshooting analytics application services, processing throughput, and message delivery

Affected alerts

IncidentBuilder Throughput IncidentBuilder Heartbeat CouchDB Client Changes Batch Processing Duration Processing Ratio Spark Batch Start Delay Gather Alert Parsing Errors Kafka MQTT Divergence

What do these alerts mean?

These alerts monitor the health of application-level services that process, correlate, and deliver analytics data. They detect situations where services have stopped, are falling behind, or are producing errors.

Processing throughput & heartbeat

IncidentBuilder Throughput fires when the processing-to-input ratio drops below 1, meaning the service is not keeping up with incoming data. IncidentBuilder Heartbeat fires when the service's streaming timestamp stops advancing, indicating the service has stalled or crashed. Processing Ratio and Batch Processing Duration monitor similar throughput and advancement metrics for the batch analytics engine.

Configuration database

CouchDB Client Changes fires when an unusually high number of change-feed requests are being made to the configuration database, which may indicate a client stuck in a reconnection loop or an unexpected burst of configuration updates.

Alert pipeline

Gather Alert Parsing Errors fires when the alert processing service encounters parsing errors, which may prevent alerts from being generated for certain metrics. Kafka MQTT Divergence fires when the rate of messages consumed from the message bus diverges significantly from the rate delivered via MQTT, indicating potential message loss in the alert delivery path.

Impact

  • Stalled processing causes delayed or missing analytics data in dashboards and reports.
  • Parsing errors in the alert pipeline may result in some alert conditions not being evaluated.
  • Message delivery divergence may mean alert notifications are not reaching their intended destinations.
  • Configuration database pressure can slow down UI operations and service configuration loading.

Investigation steps

  1. 1 Open the alert in Grafana and identify the specific service and metric that triggered the alert.
  2. 2 Check the service logs for the affected application. Look for errors, exceptions, or out-of-memory conditions.
  3. 3 Verify that upstream dependencies are healthy — a stalled processing service may be a symptom of an upstream data source or message bus issue.
  4. 4 For CouchDB alerts, check which clients are generating the change requests and whether they are behaving normally.
  5. 5 For alert pipeline issues, verify the alert processing service is running and review its logs for parsing errors or connectivity issues with the message bus.

Remediation

Stalled or slow processing

Restart the affected service after confirming upstream dependencies are healthy. If the service consistently falls behind, it may need additional resources (memory, CPU). Monitor throughput metrics after the restart to confirm recovery.

Alert pipeline errors

Review the parsing errors to determine whether they are caused by malformed input data or a service bug. If the input data format has changed, the service may need a configuration update. Restart the service after addressing the root cause.

Message delivery divergence

Check the MQTT broker health and connectivity. If the broker is healthy, the issue may be in the bridge service that translates between the message bus and MQTT. Restart the bridge service and monitor the divergence metric.

Configuration database pressure

Identify the client generating excessive change requests. A service restart may resolve a reconnection loop. If the request rate reflects legitimate activity, review the client's polling interval and consider reducing it.

Verification

  • Confirm processing throughput ratios return above 1.0 and timestamps advance.
  • Verify parsing error counts are zero.
  • Check that message delivery divergence returns to near zero.
  • Wait for the next Grafana evaluation cycle and confirm alerts return to Normal.

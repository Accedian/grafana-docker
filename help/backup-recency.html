Backup Recency Alert Guide

Troubleshooting backup schedule and retention

Affected alert

Time since last backup

What does this alert mean?

This alert fires when the elapsed time since the most recent successful database backup exceeds the configured threshold (default: 1 day). This typically indicates that a scheduled backup job did not complete.

Impact

Without a recent backup, data changes made since the last known-good snapshot are at risk. In the event of a failure or data corruption, recovery would roll back to an older point in time, potentially losing configuration changes, user data, or analytics results.

Investigation steps

  1. 1 Open the alert in Grafana and review the metric to confirm when the last successful backup was recorded. Note the gap duration.
  2. 2 Check the backup job logs — this may be a cron job, a Kubernetes CronJob, or a scheduled task depending on your deployment. Look for errors such as authentication failures, connection timeouts, or storage quota exhaustion.
  3. 3 Verify that the backup destination (object storage bucket, NFS share, or local volume) is reachable from the backup job's execution context and has sufficient available capacity.
  4. 4 If the backup job uses database credentials, confirm they have not expired or been rotated without updating the job configuration.

Remediation

Re-run the backup

Once the root cause is corrected (connectivity, credentials, quota), trigger a manual backup run to restore coverage immediately. Verify the resulting artifact is complete and matches the expected size.

Prevent recurrence

  • Add monitoring on the backup storage target to catch quota exhaustion before it blocks jobs.
  • Use alerting on the backup job exit code or duration so failures are detected within minutes, not after the recency threshold is breached.
  • Consider incremental or continuous backups to shorten the recovery point objective (RPO).

Verification

  • Confirm a fresh backup artifact exists with the current date/time.
  • Verify the artifact's integrity (e.g., checksum or test restore).
  • Wait for the next Grafana evaluation cycle and confirm the alert returns to Normal.

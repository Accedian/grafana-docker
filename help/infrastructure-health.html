Infrastructure Health Alert Guide

Troubleshooting CPU, memory, I/O, time synchronization, container scheduling, and service availability

Affected alerts

CPU Usage High High IO Wait Available Memory Docker Pending Task NTP Time Sync Dgraph Alpha Availability Prometheus NODATA

What do these alerts mean?

CPU & I/O

CPU Usage High fires when average CPU utilization exceeds the threshold for a sustained period. High IO Wait fires when processes are spending an excessive percentage of time waiting on disk I/O, indicating storage bottlenecks.

Memory

Available Memory fires when total container memory utilization drops below expected levels, suggesting containers may be failing or the system is under-provisioned.

Container scheduling

Docker Pending Task fires when container tasks have been in a pending state for an extended period. This may indicate resource shortages, missing placement constraints, or node failures preventing task scheduling.

Time synchronization

NTP Time Sync fires when hosts are not synchronized to a common time source. Time drift can cause log correlation failures, certificate validation errors, and data ordering issues across distributed services.

Service availability

Dgraph Alpha Availability fires when the expected number of Dgraph nodes is not available. Missing nodes reduce service capacity and may impact data availability.

Monitoring availability

Prometheus NODATA fires when Grafana cannot query the metrics data source. While this alert is active, other alert rules may not evaluate correctly.

Impact

  • CPU / I/O saturation slows all workloads on the affected host, including ingestion, query processing, and service health checks.
  • Memory pressure can cause out-of-memory kills, service restarts, and data loss for in-memory caches.
  • Pending tasks mean services are not running at expected capacity.
  • Time drift causes subtle data inconsistencies that are difficult to diagnose.
  • Database node loss degrades query performance and may prevent writes.
  • Monitoring outage means alerts may not fire, creating a blind spot.

Investigation steps

  1. 1 Open the alert in Grafana and identify the affected instance or node from the alert labels.
  2. 2 For CPU/IO/memory alerts, log into the host and use system tools (top, iostat, free -h) to identify the processes consuming the most resources.
  3. 3 For pending tasks, check the container orchestrator logs for scheduling errors — look for resource constraints, missing labels, or node affinity rules that cannot be satisfied.
  4. 4 For NTP alerts, verify that NTP or chrony is configured and running on the affected host. Check timedatectl or chronyc tracking for sync status.
  5. 5 For database node alerts, check the service logs on the missing node(s) for crash messages, connectivity errors, or disk space exhaustion.
  6. 6 For Prometheus NODATA, verify that the Prometheus service is running and that Grafana's data source configuration is correct.

Remediation

Resource saturation (CPU, I/O, memory)

  • Identify and address runaway processes or workloads consuming excessive resources.
  • For I/O wait, check for failing disks, filesystem fragmentation, or excessive logging.
  • If the host is genuinely under-provisioned, scale up resources.

Container scheduling

Resolve the underlying constraint preventing task placement. This may involve adding node capacity, correcting labels, or clearing a failed node from the cluster.

Time synchronization

Restart the NTP or chrony service. Ensure the NTP server is reachable and that firewall rules allow UDP port 123.

Database nodes

Restart the failed database service after resolving the root cause. Monitor the node to confirm it rejoins the cluster and begins serving queries.

Verification

  • Confirm resource utilization has returned to normal levels on affected hosts.
  • Verify all expected database nodes are online and serving traffic.
  • Check that all container tasks are running (no pending tasks).
  • Confirm NTP sync status shows synchronized on all hosts.
  • Wait for the next Grafana evaluation cycle and confirm alerts return to Normal.
